# CLAUDE.md
This file provides guidance when working with code in this repository.

## What is semipy

A runtime semiformal system. The `@semiformal` decorator and `semi()` let users express underspecified logic (natural-language conditions, extraction rules). On first invocation, an LLM generates a Python function via an **agentic pipeline** (OpenRouter + pydantic_ai with tools); the function is validated and cached. Subsequent calls reuse the cached implementation with no LLM invocation.

**Slot identity vs reuse:** `slot_id` stays unique per source line (and spec text), but `spec_equivalence_key` (in `SlotSpec`) fingerprints the durable meaning: template text, free-variable names/order, expected return type, slot category, and output names. If a new call site has no commits yet, `resolver.resolve` can **REUSE** another slot’s compiled implementation from the dispatch module when the equivalence key matches (e.g. the same `semi(f"...")` template in another notebook cell).

**REUSE vs data changes:** `spec_equivalence_key` ignores runtime values, so the same template with new inputs still resolves to **REUSE**. After REUSE, `execute_slot` runs `verify_runtime_execution` unless `runtime_input_fingerprint` on the commit matches current inputs (fingerprint is set at commit creation; it is not rewritten on each reuse, to avoid portal churn on per-row `apply`). On verify failure, resolution falls through to **ADAPT**. Verify **failures** log when `verbose` is on. A data-agnostic guard catches empty-string returns from non-empty string inputs (before the TypeAdapter branch) to force ADAPT when a reused implementation silently fails for a new input format. Another guard rejects **identity** results (output string equals single non-empty input when the input is at least 9 characters) for `function_body` / `expression` / `standalone` slots with `str` returns — this catches generated code that does `return s` on parse failure, which would otherwise pass type checks and block ADAPT when new rows are added incrementally. Open-ended NL can still pass execution checks but be wrong semantically; use `expected_type` or explicit checks where needed.

**Observation harvesting:** When slot inputs are scalar-only, `_harvest_caller_series_samples` walks the call stack to find a Series or list that contains the current value and merges distinct values into `slot.input_observation_samples` (not only before the first commit), so incremental edits to a `DataFrame` still widen the observation set for prompts. Per-call values are still recorded via `_record_slot_input_observations`.

**Branch-head resolution:** `_head_commit` (resolver) and `_get_active_commit` (store/dispatch) both use `most_recent_branch_head` from `semipy.history` to pick the newest branch head across all branches, not only `default_branch`. This ensures ADAPT commits (which go to a new branch) are used as the parent source for subsequent ADAPTs and are written to the dispatch module.

**Jupyter / IPython:** Code runs from a temp file whose basename changes every kernel restart (`.../ipykernel_*/NNNNNNNN.py`), which used to produce a new portal each time. `execute_slot` now resolves the portal anchor with `session_anchor.resolve_portal_anchor`: for paths containing `ipykernel`, the anchor is `os.getcwd()` (so the same working directory shares one portal and dispatch module). Override with `configure(session_source="/path/to/notebook.ipynb")` or env `SEMIPY_SESSION_SOURCE` when multiple notebooks share one cwd and need separate caches.

**Sketch library (pattern learning):** After GENERATE/ADAPT, binding extraction updates `sketch_library.json` under `cache_dir` (disable with `configure(sketch_library_learning=False)`). The extraction prompt in `library/binding.py` instructs the model to mark **parametric** quoted literals or identifiers when the same NL shape could repeat with different values (so a later slot can **INSTANTIATE** via substitution). `classify_with_llm` tries OpenAI Responses (`openai_model`, default `gpt-5.5` — the latest model; the generation pipeline and the contract maintainer use the same setting) first, then OpenRouter chat completions (`validator_model`) if OpenAI is missing or empty. Default `sketch_library_learning_async=False` persists sketches before `execute_slot` returns so INSTANTIATE can run in the same process; set `sketch_library_learning_async=True` for a background thread (lower latency; sketches may lag). Users do not import sketch APIs; learning is internal to `execute_slot`.

**VS Code extension:** `semipy.sessionSource` must match `configure(session_source=...)` (same resolved path string the runtime uses for `session_id`). If the workspace root is the repo, use `${workspaceFolder}/examples` when semiformal code lives under `examples/`; if the workspace root is `examples/`, use `${workspaceFolder}`. **Authoring surfaces (extension v0.3.0, `semipy-vscode/`):** the extension reads the portal JSON (incl. the newer per-slot `contract`/`ledger` and per-commit `change_record`/`commitment_record`) and surfaces it under one visual language — opacity = durability; teal = spec/contract, soft-green = intended, amber = effect/caution, red = regression/destructive; and a minimum-set rule (every chip/glyph renders only when informative, mirroring `_should_skip_key`). A single `slotInsight.computeSlotInsight` derives decision/health/contract/change/effect for all surfaces (no custom webview — native surfaces only): a **CodeLens health sentence** (`◐ ADAPT · ✓3 hold · ⚡ db://customers applied`), a hover **Explanation Card** (why/effect-diff/guarantees/touches/constraint), a **gutter health glyph**, **regressions as Problems-panel diagnostics**, **Inspect = reveal+expand the slot in the native tree** (which carries **Guarantees**/**Effects** groups with inline Relax/Revert), an interactive `#<` **Pin as contract (#>)** action (zone-tinted provenance vs effect), opacity-as-authorship dimming of the generated dispatch file, and a **steering modes** status-bar control that scaffolds `configure(...)`. The extension's data shapes live in `semipy-vscode/src/data/types.ts`; keep them in sync with `store.py` / `contract/serialize.py` / `effects/ledger.py` when serialization changes.

**Reasoning surfaces (`#<`) vs slot spec (`#>`):** For `@semiformal` functions, `lowering.scan_informal_specs` builds `SlotSpec` from contiguous `#>` comment blocks, inline `#>` on `...` statements, and `semi(...)` calls. Text on `#<` lines is **not** part of `spec_text`. Before compilation, `strip_skeleton_lines` replaces each `#<` line with a single `#` placeholder line so absolute line numbers stay aligned and `_make_slot_id` can key slots by **ordinal** inside the function (inserting or refreshing `#<` lines does not mint a new slot id). After GENERATE/ADAPT, `agents/skeleton_writer` may write new `#<` lines into the user’s source file; `#>` lines are never overwritten by that pass. Users who want to **lock in** an inference note as contract text edit `#<` into `#>` (or add a `#>` line in the same contiguous block): that changes `spec_text` and thus `spec_equivalence_key`, so the next resolution may GENERATE or ADAPT. See `examples/apache-log-usecase.md` (STAGE 7) for the promote workflow; STAGE 6 there covers pattern learning and **INSTANTIATE**. `_collect_hash_arrow_block_ranges` merges non-contiguous `#>` blocks within the same function when the gap contains no slot anchors (`name = ...` or `semi()` calls), so promoting a `#<` above code lines to `#>` folds it into the existing slot rather than creating a new one. After GENERATE/ADAPT, the skeleton writer canonicalizes indentation: move ``def`` to column 0 by subtracting its leading indent from every line; if the minimum indent of lines after ``def`` exceeds one logical level (4 spaces), reduce over-indented body lines; then prepend the on-disk ``def``-line indent to each line so the block matches the class/module column. Dispatch module sketch comments flatten multi-line `spec_template` to one line to prevent syntax errors.

**`#<` surface V2 — split-zone placement and minimum-set rules:** The skeleton writer places `#<` lines in two zones around each slot anchor line `A`. **Zone P** (provenance, above `A`) holds `goal`, `because`, `alt`, `given`; **Zone E** (effect, below `A` or below the last `#>` of a block) holds `commits`, `verified`, `yields`. Indent inherits from `A` so nested anchors (inside `for`/`if`/`with`) get correct indentation. **Minimum-set rules** (`_should_skip_key` in `steering.py`): `yields` is suppressed when the return annotation is a simple builtin (`str`, `int`, `float`, `bool`) and the generated code returns that type trivially; `given` is suppressed for single-parameter signatures; `because` is suppressed unless decision is ADAPT or there is a verify-failure context; `goal` is suppressed when the spec is already one short line; `commits` is always emitted. **`verified` is never LLM-synthesized** — `_derive_verified` derives it deterministically from `validation_result.gist_stdout` and `sample_input`. **Promotion detection** (`detect_promoted_keys` + `_detect_promoted_keys_from_file`): before rendering either zone, the writer scans the slot’s `#>` content for lines matching `#>\s*(key)\s*:`; any promoted key is omitted from `#<` (the `#>` line is the contract; a duplicate `#<` would be noise). The promoted value is stored in `advisor_state["steering_overrides"][key]` and `user_frozen=True` so subsequent synthesis does not fight the user’s promoted contract. **`spec_text` grounding:** `_build_synthesis_prompt` reads `slot_spec.spec_text` (the short on-disk `#>` text) — not `spec.prompt` (the full 500-line generation prompt) — so synthesis anchors on the correct slot spec even in large files with many slots.

**Semantic reuse check:** After verify passes on REUSE, `_should_semantic_check` evaluates whether the implementation covers observed input diversity. The check triggers when (a) the commit changed (first check for a new generation), or (b) the observation content fingerprint changed AND at least 1 REUSE call has occurred since the last check. This ensures new input patterns are evaluated promptly without excessive LLM calls on unchanging data. **Convergence cap (scale safety):** an inherently-semantic slot (triage, preference judgment, summarization) compiles to a *static* function the intent judge can keep rejecting on every fresh free-text input, which would regenerate on essentially every call (unbounded cost/latency under real, diverse traffic). `SemiConfig.semantic_verify_max_adapts` (default 2) caps semantic-recheck-driven ADAPTs per slot: `_should_semantic_check` returns False once `advisor_state["semantic_adapt_count"]` reaches the cap, so the slot converges and REUSEs thereafter. The cap is logged when hit ("Intent rechecks capped at N ...") rather than applied silently; set 0 to disable. A genuinely algorithmic slot (e.g. a propositional tautology checker that parses and enumerates assignments) generalizes, so the judge returns "satisfies intent" and the cap never engages.

**Multi-file / OOP / concurrency robustness (launch hardening):** Three classes of bug bite real (non-toy) programs and are fixed generally, not per-case. (1) **Cross-module types:** a gist subprocess runs a temp file, so only the temp dir is on `sys.path` and the user's sibling project modules (a `domain.py` the `@semiformal` code imports for its dataclasses/enums) are not importable -- generated `from my_module import T` then fails in the validation/intent-check sandboxes. `executor.subprocess_env_with_user_path` propagates the parent interpreter's `sys.path` (incl. cwd) via `PYTHONPATH` to every gist subprocess (`executor.py` and `decision.py` both use it), so imports resolve exactly as in the user's own process. Complementarily, `generator._collect_user_type_sources` now collects a type's field-annotation dependencies (e.g. an `Enum` a dataclass field references) and emits them in dependency order, so the injected sandbox source is self-contained (no `NameError`). (2) **Method receiver:** `self`/`cls` is not slot data; `lowering.scan_informal_specs` computes `receiver_names` (first param of a `Class.method` qualname named `self`/`cls`) and subtracts it from every slot's `free_variables`, keeping it out of the generated signature, the runtime data profile, and the reuse fingerprint. (3) **Concurrency / single-flight:** `slot_resolver._slot_singleflight_lock(slot_id)` gives one lock per slot so concurrent callers of the SAME slot don't each invoke the LLM (a thundering herd) or race the portal/dispatch writes -- the first generates, the rest block then REUSE; different slots stay concurrent (and the GIL already serializes a CPU-bound generated function, so the lock costs ~nothing in practice). `agents.agent._run_async` retries once on a `RuntimeError: Event loop is closed` (rebuilding the shared background loop via `_reset_async_loop`), so a submit-time race on the shared loop under heavy concurrency degrades to a retry instead of a hard crash. Verified end-to-end: 16 threads x 320 concurrent REUSE calls (0 exceptions, 0 nondeterminism, 0 corrupt portals) and 6 threads racing one uncached slot (1 generation instead of 6, 0 crashes).

**Downstream usage context:** When building a `GenerationSpec`, `build_generation_spec` reads the user's source file via `_read_user_source_for_context` and populates `user_source_code`. The LLM prompt includes this context so it can infer the expected output structure (e.g. dict keys, attribute names) from how callers consume the function's return value, even when the return type annotation is generic (like `list[dict]`).

**Behavioral contract (why/effect of changes):** Each slot carries a durable, accumulating **behavioral contract** (`semipy/contract/`) that records *why* every regeneration happened and *what its effect was*, so iterations cannot silently forget prior decisions. A `SlotContract` (persisted on `Slot.contract`, serialized in `store.py`) holds `ContractCase`s — each is an `invariant` (fixed, data-agnostic vocabulary: `non_empty`/`non_identity`/`type_match`/`category_preserving`/`idempotent`), a `metamorphic` relation (fixed registry in `relations.py`, e.g. `whitespace_invariance`), or a golden-master `example` — tagged with the `reason` it exists, the `effect` it pins, the `decision`/`origin_commit_id`, and a `status` (active/superseded/quarantined; superseded rows are kept for the audit trail). Cases are content-addressed by `(kind, input_fingerprint, assertion)` and bucketed by a **structural input fingerprint** (`contract/fingerprint.py`, digit-normalised — `03/14/2025` and `03/20/2025` share a pattern), the single source of truth that `slot_observations._obs_content_fingerprint` now delegates to. The contract is **executable**: `contract/runner.py run_contract` runs all active cases against a candidate via a rich-record gist (one subprocess; relational cases use a second pass) and maps a violated case to a validator `failure_kind` so the existing `RoutingPolicy` case-8 ADAPT route consumes it. Invariants are evaluated against the **effective output the caller consumes**, not the raw return: for a single-output STATEMENT_BLOCK the impl returns `{name: value}`, and the runner/maintainer/effect-diff project to `output[name]` (`output_names` threaded into `_build_contract_gist`) — otherwise `non_empty`/`non_identity`/`type_match` would check the dict wrapper and miss a silent empty value or an echoed input inside it. Two gate hooks in `slot_resolver.execute_slot`: the **REUSE gate** (`_run_reuse_contract_gate`, after `verify_runtime_execution` passes) forces ADAPT when a reused impl violates a carried decision (the violated case's `reason` becomes `verify_failure_context`); the **GENERATE/ADAPT gate** (`_run_generate_contract_gate`, after `SemiAgent().generate`, before `create_commit`) requires the candidate to satisfy active cases, retrying up to `contract_gate_max_retries` and quarantining the unsatisfiable remainder. **Effect tracing** (`contract/change.py compute_effect_diff`): on GENERATE/ADAPT both the parent and new impl are run over the union of contract inputs; per input *pattern* whose output changed, the diff is classified **intended** (the triggering input, or one the parent was already wrong on) vs **unintended** (a regression), recorded as a `ChangeRecord` on `Commit.change_record` (replacing the generic `message` as the real change provenance). An unintended diff fails the gate when `contract_block_regressions` is on. **Spec-change retirement (robustness):** Two cases, by how slot identity reacts. (1) Editing the `#>` NL **spec text** changes `spec_text`, which is part of `_make_slot_id` (keyed on `filename:func_qualname:spec_text`), so a **new slot** is minted with a fresh empty contract — the old slot (and its cases) is simply orphaned, so stale cases never fight the new meaning. (2) Changing the surrounding **formal code while the NL text is unchanged** — the function signature/free-variables, the return type, the slot category, or the interpolated variables of a `semi(f"...")` — keeps the same `slot_id` but changes `spec_equivalence_key`; `execute_slot` detects this (`spec_changed`) and calls `contract.access.retire_active_cases(slot, "spec changed")` before resolving, so cases derived under the old contract are quarantined and the gate/effect-diff don't fight the new intent. The maintainer re-seeds under the new spec; because cases are content-addressed, still-valid invariants (e.g. `non_empty`) reactivate the same `case_id` while now-invalid ones (e.g. a `str` `type_match` after the type became `int`) stay retired. Plain `#<` edits trigger neither path (they are not part of `spec_text`). The **maintainer** (`contract/maintainer.py maintain_contract`, scheduled by `_schedule_contract_maintenance` like sketch learning) runs after a successful GENERATE/ADAPT: it always seeds data-agnostic invariants derived from the new impl's actual behavior (so cases hold by construction — self-consistent), and when `contract_maintainer` is on, an LLM pass proposes selective golden-master examples + metamorphic relations and supersedes example cases a deliberate change broke (every proposal is verified to hold on the new source before it is added). Surfacing: the change reason/effect grounds steering synthesis (`GenerationSpec.change_summary` -> steering `by`/`unless`), `_derive_verified` appends "(N contract cases hold)", and `portal_inspect.py` prints per-slot `contract: N active / M superseded` plus the head commit's `change: reason ... | effect: +X changed, Y unintended`. SemiConfig flags: `contract_enabled` (master, default on — records contracts + change records), `contract_gate` (acceptance gate enforcement, default **off**), `contract_gate_max_retries`, `contract_block_regressions`, `contract_max_cases`, `contract_max_new_examples`, `contract_maintainer` (LLM pass, default off; deterministic seeding always runs), `contract_maintainer_async`. All read via `getattr(config, ..., default)`; the two new persisted fields load with `{}` defaults so existing portals migrate with no rewrite.

**Effects subsystem (reified real-world effects — STAGED, in progress):** Where the behavioral contract version-controls the *implementation* and the *output diff*, `semipy/effects/` is being built to make the program's **real-world effect** (on a database, file, data store, or API) a first-class, verifiable, version-controlled, provenance-tracked, revertable artifact — so an LLM cannot silently "vibe-mutate" an artifact. The design (algebraic-effects reification + event-sourced ledger + Sagas compensation + why/how/where provenance + SMT/concolic verification) lives in `/Users/r4yen/.claude/plans/steady-painting-waterfall.md`. **Stage 0 (representation) is implemented:** an *effectful* slot's generated function never touches the world — it receives an object-capability `fx` (`effects/capability.py EffectRecorder`) and **emits** a reified `EffectScript` (`list[Effect]`, fixed data-agnostic op vocab `create/read/update/delete/append/call`; `effects/models.py`); a slot is *effectful* iff its generated function declares an `fx` parameter (inferred, no new user syntax — `effects/inject.py fn_is_effectful`). `bind_slot_arguments`/`invoke_slot` gained an `extra_kwargs` passthrough (filtered to the signature, so pure slots silently ignore `fx`); `slot_resolver._call_generated_fn` injects `fx` and returns an `EffectResult` (Stage 0 is **dry-run** — nothing is applied). The validator skips return-type checks for effectful slots (the effect is recorded, not returned). `_build_user_prompt` renders an `fx`-usage block (only when `effects_enabled`) and an `## Examples` block (from active contract `example` cases, via the new `GenerationSpec.contract_examples`, populated in `build_generation_spec._examples_from_slot`). The generator action preamble seeds a recording `fx` so the model can self-test effectful candidates. Pluggable backends use a `typing.Protocol` (`effects/backends/ArtifactBackend`) + `register_artifact_backend`/`resolve_backend` (scheme-keyed, mirrors document-backend env dispatch); `backends/memory.py` (`mem://`) is the first backend (deep-copy shadow, structural-equality matching, pre-image `compensation_for` that round-trips). `SemiConfig.effects_enabled` defaults **off** (deviating from the original plan's default-on: the `fx` prompt instruction does change generation, so a pure project sees no change until opt-in); `effect_staging`/`effect_gate`/`effect_smt`/`effect_auto_apply`/etc. are reserved for later stages (all read via `getattr(config, ..., default)`). Exports from `semipy.__init__`: `Effect`, `EffectScript`, `EffectResult`, `EffectRecorder`, `ArtifactBackend`, `MemoryArtifactBackend`, `register_artifact_backend`, `resolve_backend`. **Stage 1 (shadow + static verification + gate + SQLite) is implemented:** `effects/shadow.py` `ShadowWorld` opens one shadow per target (lazily, via the backend), applies effects, serves reads, captures per-effect compensations, and commits-all/discards-all; `run_effectful_source` compiles a candidate and runs it in-process with a recorder bound to a fresh world (no subprocess gist — the fn is `fx`-confined so it cannot touch a real artifact; this deliberately drops the plan's gist-based `runner.py`). The `EffectRecorder` now binds a `world` (read-your-writes: writes apply to the shadow as recorded, so `fx.read` returns real pre-state — verified end-to-end: the model captured the real `old_tier`). `effects/verify.py verify_static` enforces the two invariants useful with zero per-slot declaration: `reversible` (every mutating effect has a backend compensation) and a `bounded_blast_radius` unbounded-guard (an `update`/`delete` must carry a selector — the "don't wipe the table" guard); each failure carries a descriptive `failure_kind`. Two gate hooks in `execute_slot` mirror the contract gates: `_run_generate_effect_gate` (stage candidate -> `verify_static` -> on violation append reason to `verify_failure_context` and regenerate up to `effect_gate_max_retries`) and `_run_reuse_effect_gate` (a reused effectful impl that violates an invariant on a new input shape forces ADAPT via `force_regenerate` -> RoutingPolicy Case 1 — so **no RoutingPolicy change was needed**, another plan simplification). `_call_generated_fn` binds a `ShadowWorld` when `effect_staging` is on (reads resolve, compensations captured) then **discards** it — Stage 1 is verified **dry-run**; real commit waits for the ledger (Stage 4). `backends/sqlite.py` (`db://`) implements the same Protocol with a real `BEGIN..ROLLBACK` transaction shadow (parameterized SQL; PRAGMA-introspected primary key; pre-image compensation), proving the abstraction generalizes from memory to a real DB with no semipy-side branching. Config (all default off): `effect_staging`, `effect_gate`, `effect_gate_max_retries`. **Stage 2 (artifact-state effect-diff / blast-radius regression) is implemented:** `effects/diff.py compute_effect_state_diff` runs the parent and the candidate over the same input against fresh shadows and compares the resulting `StateDelta`s (reusing `backend.snapshot`/`diff` — ground truth, not a script comparator). The regression rule is conservative to avoid false positives on legitimate refinements: flag only when the candidate **removes more records than the parent** (destructive escalation) or **affects materially more** (>2x and a growth larger than `effect_default_blast_radius`); a benign added history-append is not flagged. Folded into `_run_generate_effect_gate._assess` (after `verify_static`, when `effect_block_regressions` and a `parent_source` exist), so a regenerated impl that escalates the blast radius is rejected + regenerated. Only compares on the **triggering input** for now; the ledger (Stage 4) will feed previously-applied inputs into this same machinery. **Stage 3 (forall-inputs proofs) is implemented — a deliberate, documented deviation from the plan's Z3/CrossHair.** Critical-researcher finding: for the fixed selector vocabulary (AND-of-field-equalities), "prove blast radius <= 1 for all inputs" reduces to a **schema superkey lookup** (Z3 adds nothing; CrossHair reasons about function inputs not artifact cardinality, so it is the wrong tool), and neither solver is installed. The genuine, robust, dependency-free realization: `effects/schema.py` `ArtifactSchema` (unique key sets; SQLite introspects PK + UNIQUE indexes via PRAGMA, memory reports its key field) + `effects/prove.py`: `prove_bounded_blast_radius` (an `update`/`delete` is provably <=1 record iff its selector contains a unique key; `create`/`append` insert exactly one — a real forall-inputs theorem), `prove_append_only` and `prove_target_whitelist` (sound AST-structural proofs: no syntactic `fx.delete` => append-only for all inputs; all-literal targets subset of allowlist => whitelisted; computed op/target => `unknown` -> fall back to sample checks). `prove.py` exposes `ProofResult{proved|refuted|unknown}` and leaves a seam for a concolic backend (CrossHair) to refine `unknown` into a concrete-input counterexample later — no untested SAT code is shipped (honors "no stub code"). Wired into `_run_generate_effect_gate._assess` behind `effect_smt` (default off): a mutating `update`/`delete` not provably bounded by a unique key fails the gate with an actionable CEGIS message ("use a selector that includes a unique key (id)") -> regenerate. This catches `delete WHERE tier='silver'` that Stage 1 (had a selector) and Stage 2 (no parent) both pass; verified end-to-end (the model's `update WHERE id=42` is proven bounded and accepted). Config: `effect_smt` (+ reserved `effect_smt_timeout_s`/`effect_smt_max_paths` for a future concolic backend). **Stage 4 (Effect Ledger + provenance + revert + auto-apply) is implemented — the version-control spine.** `Slot.ledger` (new persisted dict field, migrates with `{}`; serialized in `store.py`) holds an append-only `EffectLedger` of `LedgerEvent`s, each keyed by `(slot_id, origin_commit_id, invocation_id)` storing the *materialized* applied effects + compensations + snapshot ref. `effects/apply.py execute_effectful` is now the single effectful runtime path (`_call_generated_fn` delegates to it, threading `slot`/`commit`/`portal` from all three call sites): it binds a `ShadowWorld`, runs with the recording `fx`, then — only when `effect_auto_apply` AND `effect_gate` AND `effect_staging` are on (hard invariant: never auto-apply an ungated effect) — **re-verifies at the real input** (the gate only saw a sample), commits the shadow to the real artifact, and appends a `LedgerEvent`; otherwise discards (dry-run). If the runtime re-verify fails it refuses loudly by raising **`EffectRefused`** (a dedicated exception, not `SemiCallError`: the function ran fine, its effect was rejected — the message states the reason + the planned-but-not-applied effect + the call site, rather than framing it as a code failure; exported from `semipy`). `effects/compensate.py revert` replays materialized compensations in reverse (`EffectResult.revert()` for in-hand undo; `revert_ledger_event(slot, event_id)` for durable undo that appends a `reverted` event — append-only). `effects/provenance.py provenance_for` walks artifact -> ledger event -> origin commit (generated source + change reason) -> slot spec; `portal_inspect.py` prints per-slot `ledger=N applied / M reverted` + the latest event's targets/commit/why. Verified end-to-end on real SQLite: generate -> gate -> auto-apply (real COMMIT) -> ledger -> provenance -> revert. Config (all default off): `effect_auto_apply`. Exports add `revert`, `provenance_for`, `SqliteArtifactBackend`. **Deferred (documented):** the learned per-slot `effect_contract`/maintainer + carried-case enforcement (risk of over-constraining; the substantive safety — reversibility + blast-radius + SMT + regression — already runs; provenance links event->commit->spec without it). **Stage 5 (externalized / irreversible effects) is implemented.** `backends/external.py ExternalArtifactBackend(sender, scheme)` is non-shadowable (`shadowable=False`) yet rides the same Protocol: `apply` *plans* (records the intent, never sends), `commit` performs the action via the user's `sender` and is **idempotent** (dedup by `payload['idempotency_key']` or `effect_id`, defending the semantic-rollback hazard), `discard` drops the plan. The `reversible` gate exempts external targets (`verify_static(..., is_external=...)`) — they are governed by approval + idempotency, not shadow-revert; the schema blast-radius proof only applies to `update`/`delete` (a `call` is opaque). `execute_effectful` adds an **approval gate**: when a script touches any non-shadowable target and `effect_require_approval_external` is on, it calls `config.effect_approval_callback(script)` with the planned (un-sent) effects ("here's what I'll do"); approve-all-or-apply-nothing (atomic, so a mixed DB+email script never commits the DB write without consent for the email); no/denied callback leaves the result **planned** (dry-run, not an error). Verified end-to-end: the model emitted `fx.call('mail://welcome', {... idempotency_key})`, the approval gate fired, the send performed once. Config: `effect_require_approval_external` (default on), `effect_approval_callback` (runtime-only). Export: `ExternalArtifactBackend`. **All six staged increments (0-5) are now implemented and tested** (107 unit tests + end-to-end runs against memory, SQLite, and an external sender with the live gpt-5.5 pipeline). **Six bugs found by adversarial end-to-end testing (the CRM demo `examples/effects_db_transactions.py`, run as a user — not the happy path) and fixed; several are general framework bugs, not effects-specific:** (1) **SQLite multi-table lock** — the backend opened a *separate connection per target*, so a customers+audit_log effect started two write transactions on one file (`database is locked`, non-atomic); fixed by sharing **one connection/transaction per shadow scope** (`_begin`/`_end`, idempotent commit/discard; schema introspection moved to a cached short-lived read conn), which also makes a multi-table effect a single atomic COMMIT. (2) **REUSE-path effect-blindness** — `verify_runtime_execution` bound the cached effectful fn without `fx` (signature mismatch -> needless regenerate); fixed by skipping the standard verify for effectful fns (the reuse effect gate owns their verification). (3) **`SemiCallError.__str__` infinite recursion** (pre-existing) — when raised without a `cause` (e.g. the auto-apply refusal) `__str__` called `str(self)` and recursed; fixed to use `super().__str__()`. (4) **Multi-line / `return semi(...)` template extraction failure** (pre-existing, affects all standalone `semi()`) — `_semi_standalone` read only one source line and `ast.parse("return semi(...)")` fails, so multi-line calls lost their variables and baked formatted data into the prompt (no reuse); fixed with `_full_call_statement` (accumulate the whole statement) + robust parse (wrap in a function body). (5) **Skeleton writer broke standalone reuse** — it rewrote the source with `#<` lines after GENERATE/ADAPT, shifting line numbers so the next call's frame lineno no longer matched the on-disk source; fixed by skipping `surface_skeleton` for `EXPRESSION_STANDALONE` slots (no `#>` block to surface there anyway). (6) **Semantic reuse check effect-blindness** — the intent judge evaluates the impl's *output*, but an effectful slot returns an `EffectScript`, so it misfired and forced ADAPT on every reuse; fixed by skipping the semantic check for effectful slots (the effect gate owns reuse verification). Demo (`examples/effects_db_transactions.py`): a CRM ETL that cleans messy records with a PURE `semi()` then commits each as an atomic multi-table DB transaction with an EFFECTFUL `semi()` — verified end-to-end: GENERATE then **REUSE** (the cached impl branches update-vs-insert from the shadow read), multi-table atomic commits, the ledger across calls, provenance, exact multi-table in-hand revert, and a blocked sweeping-delete.

## Console UX

When `configure(verbose=True)` (default), `effective_stream_display_mode(verbose=True)` uses **peek**: Rich `Live` rolling tail plus a phase strip in the terminal; Jupyter uses throttled `Panel` redraws inside a dedicated `ipywidgets.Output` (`clear_output` + one panel) so the stream does not append a long tail of panels. When `verbose=False`, stream UI is off. Peek line count, timeline, and elapsed strip are fixed in `semipy.agents.config` (`STREAM_PEEK_LINES`, `STREAM_TIMELINE`, `STREAM_SHOW_ELAPSED`). Completed model **reasoning** parts print as **Reasoning** panels in peek mode as well as the rolling tail. The Live peek is a terminal-only redraw region, so `_make_stream_sink` returns `None` when `get_console().is_terminal` is false (piped / CI / captured): there the run falls back to the plain transient lines (resolve/implement beats, tool intents, reasoning panels, receipt) instead of emitting a frozen half-rendered panel. **Persistent vs transient (console design):** the CLI narrates only the *transient process* and ends with a **receipt** that points at the editor (`Generated.` / `Reused cached implementation.` followed by a dim `-> hover the spec in your editor for why, guarantees, and effects`); the durable record (decision, why, guarantees, effect-diff, ledger) is owned by the portal and surfaced by the VS Code extension, not re-narrated in prose. Tool beats are concise human intents (e.g. `Draft the function` / `function drafted` for `execute_action_program`), never raw arg/result JSON (that is `SEMIPY_PIPELINE_TRACE`-only). Pattern-learning skips print as a quiet `  Pattern learning: ...` line, not an internal `[semipy] Sketch library:` dump.

Pipeline messages use plain language (e.g. "No reusable implementation; creating a new one.", "Implementing code...") instead of "cache miss" / "Calling agent". If a **downstream API** (e.g. Matplotlib) raises after `semi()` returns, pass `expected_type` so generation validates the shape, or reuse structured outputs from another slot instead of duplicating the same field with a second `semi()`. On the REUSE path, `slot_resolver.execute_slot` logs "Reusing cached implementation; runtime verify passed." (or "same input fingerprint") and notes "(from donor slot)" when the implementation comes from a different slot via `spec_equivalence_key` matching. Consecutive identical reuse lines for the same file and function are **batched** in `print_pipeline_log` (left column repeat count, collapsed lineno range when multiple `#>` lines differ only by line).

**ADAPT failure context:** When `verify_runtime_execution` fails and triggers ADAPT, `GenerationSpec.verify_failure_context` carries the error message from verification into the generation prompt. The ADAPT prompt in `agent.py` includes this failure reason so the LLM understands exactly what went wrong (type mismatch, execution failure, empty output, identity return) and can fix the specific issue without removing working branches.

**Slot argument binding:** `bind_slot_arguments` in `slot_call.py` maps slot `free_variables` names to the generated function's parameter names via keyword binding. When the generated function uses different names than the slot variables (common for standalone `semi()` where variables are auto-named `v0`, `v1`), the binder falls back to positional binding. The gist builder uses the same fallback when constructing test invocations.

**STATEMENT_BLOCK + schema:** For `#>` slots with a **single** output name, lowering infers `expected_type` from the enclosing method’s return annotation when the method returns that output name. If that type is **concrete** (not `Any` or plain `dict` / `dict[...]`), commit-time execution validation uses **`semipy.type_adapter.type_adapter_for`** (pydantic `TypeAdapter` with a **defining-module** namespace). Pydantic’s public `TypeAdapter.rebuild(_types_namespace=...)` still pairs that namespace with the **caller frame’s** `globals` (often `semipy.agents.validator`), which can leave dataclass schemas incomplete (`class-not-fully-defined`). `type_adapter_for` calls `_init_core_attrs` with **both** globals and locals set to the same module dict (or an explicit `globals_namespace=` for `exec` contexts). Use `type_adapter_for(T)` in user code instead of raw `TypeAdapter(T)` when validating the same dataclass types the pipeline validates. Loose `dict[...]` annotations stay permissive. The user prompt includes a short STATEMENT_BLOCK contract when strict typing applies. With **PEP 563** (`from __future__ import annotations`), `decorator._type_hints_for_lowering` uses `typing.get_type_hints` on the decorated callable so `SlotSpec.expected_type` is a resolved type, not a string forward reference (which would break `TypeAdapter` with `class-not-fully-defined` and force repeated generation). **Placeholder-output idiom:** the canonical `result = <placeholder>; #> spec; return result` shape — where the variable is initialized to a trivial placeholder (`None`/`''`/`0`/`[]`/`{}`/`set()`/`bytearray()`, or an annotation-only `result: T`) only so the skeleton parses, then read after the block — makes that variable the slot's **output**. `_infer_output_names_for_statement_block` no longer subtracts a placeholder-initialized name as "defined before" (it would otherwise return `output_names=[]`, mis-classifying the slot as a side-effect block that must `return None` and treating the output var as an *input*, which produced functions that returned None or used frame-mutation hacks). The placeholder fallback only fires when output inference is otherwise empty; a name carrying a *real* computed value before the block (e.g. `x = compute(); #> ...; return x`) stays a genuine input, not an output. **Effect-system return types** (`EffectResult`/`EffectScript`) are excluded from `_should_use_typeadapter_for_expected_type`, so an effectful slot annotated `-> EffectResult` is **not** given the STATEMENT_BLOCK dict-contract (it contradicts `return fx.script`); the effects prompt block also tells the model that when it adds `fx`, it returns `fx.script` and ignores any return-shape/output-name contract, and never hand-constructs effect types (the runtime wraps the script).

**PDF paths as slot inputs:** `execute_slot` calls `materialize_runtime_document_inputs` so existing `.pdf` paths (`Path` or string) on top-level slot kwargs or on attributes of `self` are replaced with extracted text before resolve/generate/call. Override backend with env `SEMIPY_DOCUMENT_PDF_BACKEND` (`auto`, `liteparse`, `llama_cloud`) and layout with `SEMIPY_DOCUMENT_LAYOUT_HEAVY` (`1`/`true`/`yes`). Very large PDFs still load whole text into memory; the agent tool `read_document_context` can chunk for model context during generation.

## Architecture

### Core data flow

```
@semiformal decorated function
  -> semi(f"semantic prompt with {variables}")
    -> _identify_call_site() [file:line:func -> site_id]
    -> load_portal() (cached in-memory per session_id)
    -> resolver.resolve(portal, usage, fingerprint, constants)
      -> REUSE: load function from dispatch module, optionally add ref, return compiled function
      -> ADAPT / GENERATE: build GenerationSpec (with user_source_code, enclosing_function_source)
        -> SemiAgent.generate(spec)  [agents.agent]
          -> pydantic_ai Agent (OpenRouter) with tools [agents.generator]
          -> run_stream_events(prompt, deps) -> streaming (reasoning, tool calls, response)
          -> tools: profile_data_and_flow, read_upstream_context, read_file_context, read_document_context, build_and_run_gist, validate_output
          -> extract generated_source from deps or response
        -> validate() [agents.validator] -> create_commit [history] -> save_portal -> write_dispatch_module()
        -> load and execute new function
    -> Execute function with runtime arguments
```

### Key abstractions

- **SemiCallSite**: filename, lineno, func_qualname -> site_id (SHA256).
- **PromptTemplate**: decomposed f-string with template_parts and variable_names.
- **Usage**: concrete semi() invocation; usage_id() = hash of site_id + template + constants (+ expected_type when set).
- **Decision**: REUSE, ADAPT, FORK, GENERATE, MERGE.
- **Commit**: one generated implementation (commit_id, parent_ids, generated_source, template_fingerprint, constants_snapshot, operation_signature, message, decision).
- **Slot**: per-call-site DAG (commits, branches, refs: usage_id -> commit_id); function_name_base for dispatch.
- **Portal**: per-session container (session_id, source_file, module_name, slots).
- **GenerationSpec**: prompt, call_site, template, context, expected_type, sample_input, constant_values, variable_values, user_source_code, enclosing_function_source, parent_sources, decision, ...
- **SemiAgentDeps**: spec, gist_builder, executor, generated_source, reasoning_blocks, tool_calls_log (mutable state for pydantic_ai agent run).

### Cache model (DAG versioning)

- One portal per session: `.semiformal/{session_id}.portal.json` with full DAG (slots, commits, branches, refs).
- Dispatch module: `.semiformal/runtime/{module_name}.semi.py`; **one active implementation per slot** (newest branch head across branches); compiled Python for each commit is emitted here for import.
- Resolution (simplified): match `spec_equivalence_key` for cross-slot **donor** REUSE; same slot with existing commits -> REUSE if verify passes (optionally skip verify when runtime input fingerprint matches); fingerprint or signature mismatch on a branch -> ADAPT; empty slot / no match -> GENERATE. `refs[usage_id]` short-circuits to a known commit when present.

### End-user flow (typical)

1. Author `@semiformal` code with `#>` blocks and/or `semi(f"...")`, or standalone `semi()` in normal functions.
2. First execution: `execute_slot` -> `resolver.resolve` -> GENERATE -> `SemiAgent.generate` -> validate -> `create_commit` -> save portal -> write dispatch module -> run generated function.
3. Later executions: REUSE imported function; optional `verify_runtime_execution` unless fingerprint matches; on failure, ADAPT with `verify_failure_context` in the prompt.
4. Optional: skeleton writer adds `#<` lines for traceability; user may promote lines to `#>` to change the contract.
5. Jupyter users rely on `session_anchor` (cwd for `ipykernel`); file-backed scripts use normalized source path for portal identity.

### Public API

Exports from `semipy/__init__.py`: `semiformal`, `semi`, `SemiConfig`, `configure`, `get_config`, `Decision`, `SemiCallError`, `SemiGenerationError`, `compute_spec_equivalence_key`, `register_tool`, `parse_tool_refs`, `GistBuilder`, `Gist`, `GistExecutor`, `ExecutionResult`, `SemiAgentDeps`, `ProfileDataResult`, `GistRunResult`, `OutputValidationResult`, `DocumentContextResult`, `DependencyGraph`, `SlotRef`, `DataFlow`, `attach_producer_flow`, and library helpers (`load_library`, `run_sleep_phase`, `AbstractionLibrary`, `LibraryPrimitive`, `ASTPattern`).

## Code conventions

- Use `from __future__ import annotations` and type hints in all modules.
- Use `pathlib.Path` for file I/O; normalize filenames for call-site identity.
- Implementation must be **case-independent** and **data-agnostic**: no hardcoded case-sensitive logic or data-type-specific branches.
- No placeholder or stub code; every path must be real, runnable code.
- No keyword matching or fixed pattern lists; logic driven by prompt and context.
- No emoji in code, comments, or documentation.
- LLM model references: use OpenRouter model ids (e.g. anthropic/claude-sonnet-4-6); see config.validator_model for fast validation model.
- Use Context7 MCP for library/API documentation when needed.
- Prefer existing dependencies; introduce new ones only with user awareness.
- **Type checking**: Generated function return values are validated with `isinstance`; when pydantic is available, prefer `semipy.type_adapter.type_adapter_for(expected_type)` over raw `TypeAdapter` so namespaces match the defining module (see STATEMENT_BLOCK section above).
- When testing the code, act an user and actually using theagentic tool you builtin to run, inspect ande debug; the goal is not to see if the code runs, but to see if the output is what you expected.
